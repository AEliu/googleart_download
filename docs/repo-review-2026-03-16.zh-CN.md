# 仓库审查建议（2026-03-16）

> 范围：`googleart_download` 全仓库，基于当前主干代码、测试与文档状态。

## 总体评价

该项目已经具备较成熟的工程化基础：

- 模块边界清晰（`cli` / `download` / `batch` / `metadata` / `reporting`）
- 单元测试覆盖面较广，关键流程已有回归
- 文档体系完整（使用说明、架构、ADR、发布记录）
- CI/本地质量门槛明确（`ruff`、`mypy`、`unittest`）

## 建议清单（按优先级）

### P1：抽象 sync/async HTTP 重试公共逻辑，减少重复维护成本

当前 `HttpClient` 与 `AsyncHttpClient` 各自实现了近似的 `_request_with_retries`、错误映射与重试日志逻辑。建议提炼出共享策略层（例如纯函数 + 回调），把 sync/async 差异收敛到“请求执行器”与“sleep 实现”。

**收益：**

- 降低行为漂移风险（例如某次只改了 sync 分支）
- 更容易新增指标上报（重试次数、错误分类）
- 让 transport 调优更集中

### P1：补齐“真实网络行为”层的集成测试策略

当前测试以单元与流程拼装验证为主，速度很快、反馈很好。但针对真实网页结构变化、短链解析、代理行为等外部契约，建议增加“可录制/可回放”的契约型集成测试（例如 VCR/fixture snapshot）并在 CI 中做轻量抽样。

**收益：**

- 更早发现 Google 页面结构变化带来的回归
- 减少“本地全绿但线上失败”的概率
- 与现有 smoke workflow 形成互补

### P2：减少 `cli.main` 中函数体内动态导入，提升可读性与可分析性

`cli/main.py` 中存在多处函数体内 `from . import ...`。若非循环依赖刚需，建议转为模块级显式导入；若存在循环依赖，可考虑下沉共享类型/服务层来断环。

**收益：**

- 提升静态分析与可导航性
- 降低维护者理解成本
- 让依赖关系更透明

### P2：增加最小基准测试（benchmark）以支撑 transport 和 stitch 调优

项目已在文档中规划 transport 调优与运行指标。建议增加一个小型 benchmark 脚本/测试集，固定输入与网络条件，输出关键指标（每秒 tile、失败重试比、总耗时、峰值内存）。

**收益：**

- 避免“优化体感化”
- 为 `--workers`、超时、连接池策略提供量化依据
- 让 PR 性能回归可检测

### P3：重新评估 Python 版本门槛（目前 3.13+）

当前版本要求 `>=3.13`。如果没有强依赖 3.13 专属特性，可考虑在后续版本评估是否兼容到 3.11/3.12，以降低使用门槛并扩大可安装范围。

**收益：**

- 提升潜在用户覆盖
- 降低环境搭建阻力

### P3：把“项目状态中的 TODO”拆分成可执行 issue 模板

`docs/project-status.md` 中 TODO 信息丰富，但较长。建议在仓库 issue 模板中沉淀为“问题定义/验收标准/风险点”，并标注 owner 与里程碑。

**收益：**

- 减少路线图与实际执行脱节
- 提升协作效率和可追踪性

## 建议落地顺序（30 天）

1. 先做 P1-1（HTTP 重试抽象）+ 回归测试，避免后续功能叠加时继续复制逻辑。
2. 并行推进 P1-2（契约型集成测试最小集合，先覆盖短链解析和 metadata-only）。
3. 再做 P2-1（CLI 导入结构梳理），确保改动小步且可回退。
4. 最后视发布节奏决定 P3（Python 版本策略、Issue 模板化）。
